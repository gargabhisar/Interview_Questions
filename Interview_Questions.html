MongoDB vs SQL Databases

SQL databases are relational, schema-first systems that store data in tables with enforced relationships and ACID transactions. MongoDB is a document-oriented NoSQL database that stores flexible JSON-like BSON documents in collections, optimized for horizontal scaling and rapid schema evolution.

On SQL Server you also tune parallelism with MAXDOP (maximum degree of parallelism), which limits how many CPUs a single query may useâ€”separate from the SQL vs document-store choice but important for relational performance.

AspectSQLMongoDB
Data modelTables, rows, columnsCollections, documents
SchemaFixed, enforcedFlexible, optional
ScalingMostly verticalHorizontal sharding
JoinsNative JOINsEmbedding or $lookup
-- SQL: normalized order query
SELECT o.OrderId, c.Name
FROM Orders o
JOIN Customers c ON o.CustomerId = c.Id;

-- MongoDB: embedded or lookup
db.orders.aggregate([
  { $lookup: { from: "customers", localField: "customerId",
      foreignField: "_id", as: "customer" } }
]);

Key Points

  • Choose SQL when you need strong consistency, complex joins, and reporting.
  • Choose MongoDB for flexible schemas, high write throughput, and document-centric apps.
  • Many teams use both: SQL for transactional core, MongoDB for logs or content.

Interview Answer

SQL is relational and ACID-focused with strict schemas; MongoDB is document-based and schema-flexible, better for scale-out and evolving data shapes. I pick based on consistency needs, query patterns, and team expertiseâ€”not hype.

Monolithic vs Microservices Architecture

A monolith is a single deployable application containing all modulesâ€”UI, business logic, and data accessâ€”sharing one process and often one database. Microservices split the system into independently deployable services, each owning a bounded context and communicating over HTTP or messaging.

Monoliths are simpler to develop and debug early on. Microservices improve scalability and team autonomy at the cost of distributed complexityâ€”network failures, observability, and deployment pipelines multiply.

AspectMonolithMicroservices
DeploymentSingle unitIndependent services
ComplexityLower initiallyHigher operational overhead
ScalingScale entire appScale hot services
Best whenSmall teams, MVPsLarge domains, many teams

Key Points

  • Start monolith unless clear boundaries justify splitting.
  • Microservices need strong DevOps and observability.
  • Domain boundaries matter more than technology choice.

Interview Answer

A monolith is one deployable app that is easy to build initially but harder to scale independently. Microservices decompose by business capability for independent deployment and scaling, but add network, consistency, and operations complexity. I choose based on team size, domain clarity, and scaling needsâ€”not hype.

Multiple catch Blocks in try-catch

C# evaluates catch clauses top to bottom; the first matching exception type handles it. Specific exceptions must come before general ones. Only one catch runs per thrown exception.

try
{
    var json = File.ReadAllText(path);
    var data = JsonSerializer.Deserialize<Order>(json);
}
catch (JsonException ex)
{
    _logger.LogWarning(ex, "Invalid JSON");
}
catch (FileNotFoundException ex)
{
    _logger.LogError(ex, "File missing");
}
catch (IOException ex) when (ex.HResult == -2147024784)
{
    // filter clause â€” disk full
}
catch (Exception ex)
{
    _logger.LogCritical(ex, "Unexpected failure");
    throw; // preserve stack trace
}

Key Points

  • Order: most specific â†’ least specific; Exception last.
  • catch when filters on type plus condition without catching base first incorrectly.
  • Use throw; not throw ex; to rethrow preserving stack.
  • finally runs regardlessâ€”use for cleanup (Dispose patterns).

Interview Answer

Multiple catch blocks let you handle different failure types differently. Place specific handlers first, optionally use exception filters with when, and use a general catch only for logging before rethrow or graceful degradation.

Database Normalization

Normalization organizes data to reduce redundancy and update anomalies by decomposing tables according to normal forms (1NF through BCNF). Each step removes a specific kind of dependency problem.

Well-normalized OLTP schemas prevent insert/update/delete anomalies; selective denormalization is acceptable only when profiling proves read bottlenecks.

FormRule
1NFAtomic values, no repeating groups
2NF1NF + no partial dependency on composite key
3NF2NF + no transitive dependency on non-key columns
-- Before (redundant):
Orders(OrderId, CustomerName, CustomerCity, Product, Qty)

-- After 3NF:
Customers(CustomerId, Name, City)
Products(ProductId, Name)
Orders(OrderId, CustomerId, ProductId, Qty)

Key Points

  • Normalization improves data integrity and reduces duplicate storage.
  • Over-normalization can hurt read performanceâ€”denormalize selectively for hot paths.
  • Interviewers often want you to spot redundancy and propose split tables.

Interview Answer

Normalization splits tables to eliminate redundancy and anomalies. I apply 1NFâ€“3NF in OLTP design, then denormalize only where measured read performance requires it.

Boyce-Codd Normal Form (BCNF)

BCNF is a stricter form of 3NF: every determinant must be a candidate key. A table is in BCNF when for every functional dependency X â†’ Y, X is a superkey. It eliminates anomalies 3NF might still allow when multiple overlapping candidate keys exist.

When a non-key attribute determines another non-key attribute, decompose into separate tables while keeping joins lossless and preserving dependencies where possible.

-- Violates BCNF (Teacher â†’ Room, but Teacher is not a key):
Enrollment(Student, Course, Teacher, Room)

-- Fix: separate Teacher â†’ Room dependency
TeacherRoom(Teacher, Room)
Enrollment(Student, Course, Teacher)

Key Points

  • BCNF âŠ‚ 3NF; all BCNF tables are 3NF, not vice versa.
  • Violations occur when non-key attributes determine other non-key attributes.
  • Decomposition must be lossless and preserve dependencies where possible.

Interview Answer

BCNF requires every determinant to be a candidate key. If Teacher determines Room but Teacher isn't a key, I decompose into separate tables to remove the anomaly while keeping joins lossless.

Database Indexes

Indexes are auxiliary data structures (typically B-trees) that speed up data retrieval by avoiding full table scans. They trade faster reads for extra storage and slower writes due to index maintenance.

Design indexes around real query patternsâ€”filters, joins, ORDER BY, and GROUP BYâ€”and validate with execution plans rather than indexing every column.

-- Clustered index on primary key (default in SQL Server)
CREATE CLUSTERED INDEX IX_Emp_Id ON Employees(EmployeeId);

-- Nonclustered index for frequent filter
CREATE NONCLUSTERED INDEX IX_Emp_Dept
ON Employees(DepartmentId) INCLUDE (LastName, Salary);
ProsCons
Faster SELECT/WHERE/JOINSlower INSERT/UPDATE/DELETE
Enforces uniqueness (unique index)Extra disk and memory

Key Points

  • Index columns used in WHERE, JOIN, ORDER BY, and GROUP BY.
  • Too many indexes hurt write-heavy workloads.
  • Use covering indexes (INCLUDE) to avoid key lookups.

Interview Answer

Indexes accelerate lookups like a book index but slow writes. I index selective, frequently queried columns, verify with execution plans, and avoid over-indexing OLTP tables.

Clustered vs Non-Clustered Indexes

A clustered index defines the physical sort order of table rowsâ€”there can be only one per table. A non-clustered index is a separate structure with pointers (or row locators) back to the data.

Range scans on the clustered key are very efficient; non-clustered indexes may require key lookups when columns aren't covered.

ClusteredNon-Clustered
One per tableMany allowed

  EmployeeId INT PRIMARY KEY,  -- clustered by default
  DeptId INT,

);
CREATE NONCLUSTERED INDEX IX_Dept ON Employees(DeptId);

Key Points

  • Range scans on clustered key are very efficient.
  • Non-clustered indexes may require key lookups (bookmark/RID).
  • Choose clustered key on narrow, ever-increasing column (e.g., identity).

Interview Answer

Clustered index sorts the table physicallyâ€”one per table. Non-clustered indexes are separate lookup structures. I put the clustered index on the most common range-scan column, usually the PK.

SQL Execution Plans

An execution plan shows how the optimizer retrieves data: scan vs seek, join algorithms, sort/hash operations, and estimated vs actual row counts. Reading plans is essential for performance tuning.

Compare estimated vs actual rows; large gaps often mean stale statistics or parameter sniffing issues.

SET STATISTICS TIME ON;

FROM Employees e
INNER JOIN Departments d ON e.DeptId = d.DeptId
WHERE e.Salary > 80000;

-- View plan in SSMS: Ctrl+M or SET SHOWPLAN_ALL ON;

Key Points

  • Look for table/index scans on large tablesâ€”often fixable with indexes.
  • Compare estimated vs actual rows; big gaps suggest stale statistics.
  • Watch for implicit conversions, key lookups, and expensive sorts.

Interview Answer

I use execution plans to see whether SQL Server scans or seeks, which joins it picks, and where cost concentrates. Red flags are scans on big tables, bad cardinality estimates, and missing indexes.

UNION combines result sets from two or more SELECT statements and removes duplicate rows (distinct sort/hash). UNION ALL concatenates all rows including duplicatesâ€”faster when duplicates are acceptable or impossible.

UNION merges queries and removes duplicates; UNION ALL just stacks them. I use UNION ALL for performance unless I explicitly need distinct combined results.

ISNULL vs COALESCE

Both replace NULL with a value. ISNULL is SQL Serverâ€“specific, takes exactly two arguments, and uses the first argument's data type. COALESCE is ANSI-standard, accepts multiple arguments, and returns the first non-NULL value.

ISNULL can silently truncate if types differ; COALESCE follows data-type precedence rulesâ€”prefer COALESCE for portability.

SELECT
  ISNULL(MiddleName, 'N/A') AS Middle,
  COALESCE(MiddleName, NickName, 'N/A') AS Display
FROM Employees;

SELECT COALESCE(NullableInt, 0) FROM T;

Key Points

  • COALESCE evaluates arguments until first non-NULL; ISNULL evaluates once.
  • ISNULL can truncate if types differ; COALESCE follows precedence rules.
  • Prefer COALESCE for portability and multiple fallbacks.

Interview Answer

ISNULL replaces NULL with one fallback and is SQL Serverâ€“specific. COALESCE is standard, supports multiple fallbacks, and is my default unless I need ISNULL's two-arg simplicity in T-SQL.

Tables vs Views

Tables hold data; views are stored queries that expose a simplified or secured interface. I use views to encapsulate joins and permissions, but the underlying tables do the real storage.

Stored procedures are precompiled batches for business logic, DML, and orchestration. Functions return a single scalar or table value and are designed for reuse inside queriesâ€”with restrictions on side effects.

Scalar UDFs can hurt performance on large sets; inline table-valued functions are preferred for set-based operations.

  • SPs for transactions and workflows; functions for reusable calculations.
  • Scalar UDFs can hurt performance on large setsâ€”inline TVFs preferred.
  • Functions cannot modify persistent data in SQL Server.

Interview Answer

Stored procedures execute business logic and can modify data; functions return computed values for use in queries. I use SPs for operations and functions for pure calculations embedded in SELECTs.

Common Table Expressions (CTE)

A CTE is a temporary named result set defined with WITH, scoped to a single statement. It improves readability for recursive queries and multi-step transformations without creating permanent objects.

Recursive CTEs traverse hierarchies (org charts, BOMs). The optimizer may inline CTEs rather than materialize them.

WITH DeptTotals AS (
  SELECT DeptId, SUM(Salary) AS TotalSalary
  FROM Employees
  GROUP BY DeptId
)
SELECT d.DeptName, t.TotalSalary
FROM DeptTotals t
JOIN Departments d ON t.DeptId = d.DeptId;

Key Points

  • Recursive CTEs traverse hierarchies (org charts, BOMs).
  • CTEs are not guaranteed to be materializedâ€”optimizer may inline.
  • Can reference multiple CTEs in one WITH clause.

Interview Answer

A CTE is a named subquery using WITH that makes complex SQL readable and supports recursion. It's scoped to one statement and doesn't persist like a temp table unless the optimizer materializes it.

Use table variables for small staging; temp tables for joins on large intermediate sets.

  • Use table variables for small staging (< few thousand rows).
  • Temp tables for joins on large intermediate sets.

Temp tables support indexes and accurate stats for larger data; table variables are lightweight for small in-memory staging. Wrong choice causes bad plansâ€”I've moved hot paths from @vars to #temp when row counts grew.

Subquery vs CTE

Both nest queries, but CTEs name intermediate results at the top for clarity and recursion. Correlated subqueries run per outer row and can be slower; CTEs read cleaner for multi-step logic.

Performance is often identicalâ€”the optimizer may produce the same plan. Prefer EXISTS over IN for correlated checks on large tables.

SELECT Name FROM Employees
WHERE DeptId IN (SELECT DeptId FROM Departments WHERE Region = 'West');

-- CTE (clearer for multiple references)
WITH WestDepts AS (
  SELECT DeptId FROM Departments WHERE Region = 'West'
)
SELECT e.Name FROM Employees e

Deadlocks

A deadlock occurs when two or more sessions hold locks each other needs, forming a cycle. SQL Server detects deadlocks, picks a victim, rolls back its transaction, and returns error 1205.

Access resources in consistent order across transactions and keep transactions short.

-- Session A                    -- Session B
BEGIN TRAN                      BEGIN TRAN
UPDATE Accounts SET ... Id=1    UPDATE Accounts SET ... Id=2
UPDATE Accounts SET ... Id=2    UPDATE Accounts SET ... Id=1
-- deadlock: one victim rolled back

Key Points

  • Access resources in consistent order across transactions.
  • Keep transactions short; avoid user interaction inside transactions.
  • Use appropriate isolation; retry victim transactions in app code.

Interview Answer

Deadlocks are circular lock waitsâ€”SQL Server kills one session. I prevent them by consistent lock ordering, short transactions, proper indexes to reduce lock time, and retry logic for 1205 errors.

NOLOCK Hint (READ UNCOMMITTED)

WITH (NOLOCK) or READ UNCOMMITTED allows reading uncommitted dataâ€”dirty reads, non-repeatable reads, and phantom rows are possible. It reduces blocking but sacrifices consistency.

Prefer READ COMMITTED SNAPSHOT for read concurrency without dirty reads on critical data.

WHERE OrderDate >= '2024-01-01';

-- Equivalent session setting:

Key Points

  • Never use NOLOCK on financial or authoritative reporting data.
  • Can read rows that roll back or see inconsistent page states.
  • Prefer READ COMMITTED SNAPSHOT for read concurrency without dirty reads.

Interview Answer

NOLOCK reads without shared locks, allowing dirty reads and inconsistencies. I avoid it for critical data; if I need read concurrency, I use RCSI or snapshot isolation instead.

SQL Transactions

A transaction groups one or more operations into an atomic unit following ACID: Atomicity, Consistency, Isolation, Durability. Either all changes commit or all roll back.

Use TRY/CATCH with XACT_ABORT ON in modern T-SQL; keep transactions short to reduce lock duration and deadlocks.

BEGIN TRANSACTION;
  UPDATE Accounts SET Balance = Balance - 100 WHERE Id = 1;
  UPDATE Accounts SET Balance = Balance + 100 WHERE Id = 2;
  IF @@ERROR <> 0 ROLLBACK TRANSACTION;
  ELSE COMMIT TRANSACTION;

Key Points

  • Keep transactions short to reduce lock duration and deadlocks.
  • Use TRY/CATCH with XACT_ABORT ON in modern T-SQL.
  • Nested transactions use @@TRANCOUNT; savepoints allow partial rollback.

Interview Answer

Transactions ensure ACIDâ€”either all SQL in the batch succeeds or none does. I wrap related DML in explicit transactions, handle errors with rollback, and keep them as short as possible.

MERGE Statement

MERGE performs INSERT, UPDATE, or DELETE in one statement based on a join between target and source. Useful for upserts and synchronizing staging to production tables.

WHEN NOT MATCHED THEN INSERT (ProductId, Price)
  VALUES (s.ProductId, s.Price)
WHEN NOT MATCHED BY SOURCE THEN DELETE;

Key Points

  • Requires proper indexes on join keys for performance.
  • Known concurrency issuesâ€”consider separate INSERT/UPDATE in high contention.
  • OUTPUT clause can capture merged rows for auditing.

Interview Answer

MERGE upserts and syncs target tables from a source in one statement. I use it for ETL staging loads but watch for locking and known race conditions under heavy concurrency.

How Claims Work in JWT

A claim is a statement about the subject (user or client), stored as a name–value pair in the JWT payload (middle part). After Base64URL decode, the payload is JSON. The server (or identity provider) puts claims in when issuing the token; the API reads them after signature and lifetime validation.

JWT structure (three parts)

  1. Header — algorithm and type (alg, typ).
  2. Payload — claims (who, what roles, when it expires).
  3. Signature — proves the payload was not tampered with.

Example payload (claims):

{
  "sub": "user-42",
  "email": "user@example.com",
  "role": "Admin",
  "iss": "https://my-api.com",
  "aud": "https://my-api.com",
  "exp": 1717000000,
  "iat": 1716996400
}

CORS in ASP.NET Core

CORS (Cross-Origin Resource Sharing) is a browser security feature. It blocks JavaScript on one origin (scheme + host + port) from calling an API on another origin unless the API returns allowed CORS headers.

Why is CORS required?

Example: React app at http://localhost:3000 calls API at https://localhost:5001 — different origins. Without CORS headers the browser blocks the response even if the API returned 200.

Note: Postman, server-to-server, and mobile native apps are not restricted by CORS — only browsers.

How to configure in ASP.NET Core

// 1. Register policy
builder.Services.AddCors(options =>
{
    options.AddPolicy("WebApp", policy =>
        policy.WithOrigins("http://localhost:3000", "https://app.example.com")
              .AllowAnyHeader()
              .AllowAnyMethod()
              .AllowCredentials()); // only with specific origins, not AllowAnyOrigin
});

var app = builder.Build();

// 2. Apply in pipeline (before MapControllers / auth as needed)
app.UseCors("WebApp");

app.MapControllers();

Task.Run Explained

Task.Run queues a delegate on the .NET thread pool for CPU-bound work. The caller gets a Task immediately; pool threads execute the delegate concurrently. It is ideal when synchronous, compute-heavy code would block a sensitive thread (UI, ASP.NET request thread in older patterns).

// Offload CPU work

// Overload with CancellationToken
var cts = new CancellationTokenSource(TimeSpan.FromSeconds(30));
await Task.Run(() => ProcessBatch(items, cts.Token), cts.Token);

// Long-running alternative when you need a dedicated thread
await Task.Factory.StartNew(
    () => LongRunningListener(),
    TaskCreationOptions.LongRunning);

Key Points

  • Schedules work to ThreadPoolâ€”not for I/O-bound async APIs.
  • Returns hot Task; exceptions stored on Task, surfaced on await/Wait.
  • Prefer async I/O APIs over Task.Run for network/database calls.
  • LongRunning creates a non-pool thread when pool starvation is a risk.

Interview Answer

Task.Run moves CPU-bound synchronous work to a background thread-pool thread and returns a Task you can await. Use it for parallel computation, not as a substitute for truly asynchronous I/O.

Thread vs Task

A Thread is an OS-level execution unitâ€”you create it, set priority, and manage its lifetime manually. A Task is a higher-level abstraction representing work that the thread pool may run; the runtime schedules Tasks onto available threads efficiently.

// Thread â€” explicit, heavier

thread.Start();
thread.Join();

// Task â€” preferred for most scenarios
await Task.Run(() => DoWork());

// Task enables composition
var tasks = urls.Select(url => DownloadAsync(url));
await Task.WhenAll(tasks);

Key Points

  • Threads are expensive (~1 MB stack); pool reuse is cheaper.
  • Tasks support await, WhenAll, cancellation, and exception aggregation.
  • Creating many raw Threads can hurt scalability; Task uses pool.
  • Use dedicated Thread only for STA COM, long-blocking listeners, or special cases.

Interview Answer

Threads are low-level OS workers you manage directly. Tasks represent units of work scheduled by the thread pool with rich async composition. In modern .NET, prefer Task/async unless you have a specific reason to own a Thread.

Task.Wait and Blocking Pitfalls

Task.Wait(), Task.Result, and GetAwaiter().GetResult() synchronously block the calling thread until the Task completes. On contexts with a synchronization context (UI, legacy ASP.NET), blocking while waiting for code that needs that same context causes deadlocks.

// Dangerous on UI / ASP.NET (deadlock risk)

{

}

// Preferred

{
    return await _service.GetOrderAsync(id);
}

// Library code avoiding context capture

Key Points

  • Never block on async code in ASP.NET Core request paths if avoidable.
  • AggregateException wraps Task exceptions on Wait/Resultâ€”harder to debug.
  • async all the way down is the scalable pattern.
  • ConfigureAwait(false) in libraries reduces deadlock on sync-over-async.

Interview Answer

Task.Wait blocks the current thread until completion. Avoid it in async-friendly appsâ€”use await instead. Blocking on incomplete Tasks tied to the same synchronization context is a common deadlock source.

Alternatives to Task.WhenAll

Task.WhenAll waits for multiple Tasks concurrently. Alternatives depend on semantics: fail-fast vs collect-all-errors, throttling, sequential dependency, or parallel loops over CPU work.

// WhenAny â€” first completion (timeout/race pattern)

// Parallel.ForEachAsync â€” bounded parallelism (.NET 6+)

    new ParallelOptions { MaxDegreeOfParallelism = 4 },
    async (item, ct) => await ProcessAsync(item, ct));

// Manual throttling with SemaphoreSlim
using var gate = new SemaphoreSlim(4);
var throttled = items.Select(async i => {
    await gate.WaitAsync();
    try { return await WorkAsync(i); }
    finally { gate.Release(); }
});
await Task.WhenAll(throttled);

Key Points

  • WhenAny for timeouts, caching first response, or competitive calls.
  • Parallel.ForEach/ForEachAsync for CPU-bound batch with degree control.
  • SemaphoreSlim limits concurrent I/O when APIs have rate limits.
  • Task.WhenAll fails on first fault unless you observe all Task.Exception.

Interview Answer

Use WhenAll for unconstrained parallel awaits. Use WhenAny for first-wins scenarios, Parallel.ForEachAsync for bounded CPU parallelism, and SemaphoreSlim when you must cap concurrent operations. Choose based on error handling and resource limits.

  • volatile/Interlocked address visibility and atomic ops, not complex invariants.

Multithreading executes concurrent paths in one process for parallelism. Success depends on identifying shared state, choosing correct synchronization, and avoiding deadlocks and racesâ€”often by immutability or concurrent collections.

DbContext is not thread-safe and tracks entity changes for a unit of work. Registering it as scoped gives each HTTP request its own context instance, aligning with one logical transaction per request and preventing cross-request state leakage.

Constructor Chaining in C#

Constructor chaining lets one constructor call another in the same class (this) or a base class constructor (base). This avoids duplicating initialization logic and ensures objects are always created in a valid state.

The called constructor runs first. Use : this(...) to forward to another constructor in the same type, and : base(...) to invoke a parent constructor before your own body executes.

public class Employee
{
    public int Id { get; }
    public string Name { get; }

    public Employee(int id) : this(id, "Unknown") { }

    public Employee(int id, string name)
    {
        Id = id;
        Name = name;
    }
}

public class Manager : Employee
{
    public Manager(int id, string name) : base(id, name) { }
}

Key Points

  • this(...) chains constructors within the same class.
  • base(...) calls a base class constructor.
  • Chaining must be the first statement in the constructor header.

Interview Answer

Constructor chaining reuses initialization through this or base calls so setup logic is not duplicated. The chained constructor always runs first, which helps enforce consistent object construction across overloads and inheritance.

Why async/await When Task Already Exists?

Task and Task<T> represent ongoing or completed asynchronous work. Raw Tasks force manual continuation chaining (.ContinueWith, callbacks), exception handling, and return-value plumbing. async/await is syntactic sugar that compiles to a state machine: readable sequential code, automatic exception propagation, and correct Task return types for callers.

// Without async/await â€” hard to read and error-prone
Task<Order> GetOrderOld(int id)
{
    return _repo.GetCustomerAsync(id)
        .ContinueWith(t =>
        {
            if (t.IsFaulted) throw t.Exception!;
            return _repo.GetOrderAsync(t.Result.OrderId);
        }).Unwrap();
}

// With async/await â€” same semantics, clearer flow
async Task<Order> GetOrderAsync(int id)
{
    var customer = await _repo.GetCustomerAsync(id);
    return await _repo.GetOrderAsync(customer.OrderId);
}

Key Points

  • Task = the abstraction; async/await = ergonomic way to compose Tasks.
  • Await unwraps exceptions; ContinueWith requires manual fault checking.
  • Compiler generates IAsyncStateMachineâ€”no magic threads, just continuations.
  • Methods marked async always return Task/Task<T> (or ValueTask) to callers.

Interview Answer

We still use Taskâ€”async/await does not replace it. Tasks model async operations; async/await lets us write and maintain that logic like synchronous code while preserving non-blocking behavior and proper exception flow.

A class (or inline delegate) you write to add cross-cutting logic: correlation IDs, request timing, API keys, global exception JSON responses.

{
    private readonly RequestDelegate _next;
    public RequestTimingMiddleware(RequestDelegate next) => _next = next;

    public async Task InvokeAsync(HttpContext context)
    {

In ASP.NET Core, Run(), Use(), and Map() are middleware extension methods used to build the HTTP request pipeline.

Each middleware can handle the request, pass it to the next middleware, or stop the pipeline.

{
    await context.Response.WriteAsync("First Run");
});

{
    await next(); // NEVER executes for requests handled above
});

Analogy: Road closed here â€” no request moves further.

{
    await context.Response.WriteAsync("Middleware 1 Start\n");
    await next();
    await context.Response.WriteAsync("Middleware 1 End\n");
});

app.Run(async context =>
{
    await context.Response.WriteAsync("Run Middleware\n");
});

Middleware 1 End

{

});

Custom Exception Middleware

public class ExceptionMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<ExceptionMiddleware> _logger;

    public ExceptionMiddleware(RequestDelegate next, ILogger<ExceptionMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task Invoke(HttpContext context)
    {
        try
        {
            await _next(context);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Unhandled exception occurred");
            await HandleException(context, ex);
        }
    }

    private static async Task HandleException(HttpContext context, Exception ex)
    {
        context.Response.ContentType = "application/json";

        var (statusCode, message) = ex switch
        {
            KeyNotFoundException => (StatusCodes.Status404NotFound, ex.Message),
            UnauthorizedAccessException => (StatusCodes.Status401Unauthorized, ex.Message),
            ArgumentException => (StatusCodes.Status400BadRequest, ex.Message),
            NotImplementedException => (StatusCodes.Status501NotImplemented, ex.Message),
            _ => (StatusCodes.Status500InternalServerError, "Internal server error")
        };

        context.Response.StatusCode = statusCode;

        await context.Response.WriteAsJsonAsync(new
        {
            success = false,
            message,
            statusCode
        });
    }
}

Resiliency and Polly in ASP.NET Core

Resiliency means your app keeps working (or fails gracefully) when dependencies are slow, down, or flaky. For HTTP calls between services, you handle transient faults (timeouts, 503, network blips) without taking down the whole system.

Polly is a .NET resilience library: it defines policies (retry, circuit breaker, timeout, etc.) and runs your code through them. With IHttpClientFactory, policies attach to named or typed HttpClient registrations.

Transient vs permanent failures

Retry?Examples
Yes (with limits)408, 429, 502, 503, 504, socket timeout, connection reset
No400 validation, 401/403 auth, 404 not found, business rule errors

Only retry idempotent operations unless you have deduplication (safe: GET; risky: POST that creates orders).

Common Polly policies

PolicyPurpose
RetryRe-attempt failed calls (often with exponential backoff)
Circuit breakerStop calling a failing service; fail fast; probe recovery later
TimeoutCancel calls that run too long
BulkheadLimit concurrent calls so one dependency cannot exhaust threads
FallbackReturn cached/default response when dependency fails

Circuit breaker states

  • Closed — normal; calls pass through; failures are counted.
  • Open — too many failures; calls fail immediately (no load on sick service).
  • Half-open — after break duration, one trial call; success closes circuit, failure reopens.

HttpClient + Polly (classic integration)

NuGet: Microsoft.Extensions.Http.Polly (wraps Polly for HTTP).

builder.Services.AddHttpClient<IPaymentClient, PaymentClient>(client =>
{
    client.BaseAddress = new Uri(builder.Configuration["Services:PaymentApi"]!);
    client.Timeout = TimeSpan.FromSeconds(30);
})
.AddTransientHttpErrorPolicy(policy =>
    policy.WaitAndRetryAsync(
        3,
        retryAttempt => TimeSpan.FromSeconds(Math.Pow(2, retryAttempt)),
        onRetry: (outcome, delay, attempt, ctx) =>
        {
            // log attempt, status, exception
        }))
.AddTransientHttpErrorPolicy(policy =>
    policy.CircuitBreakerAsync(
        handledEventsAllowedBeforeBreaking: 5,
        durationOfBreak: TimeSpan.FromSeconds(30)));

AddTransientHttpErrorPolicy retries HTTP 5xx, 408, and HttpRequestException (not 4xx client errors).

Combining policies (order matters)

Typical outer-to-inner order for HTTP: Timeout → Circuit breaker → Retry → actual request. Timeouts should bound total wait including retries.

// Manual policy wrap (conceptual)
var pipeline = Policy.WrapAsync(retryPolicy, circuitBreakerPolicy, timeoutPolicy);
await pipeline.ExecuteAsync(() => httpClient.GetAsync("/orders"));

.NET 8+ built-in resilience (alternative)

Microsoft also ships Microsoft.Extensions.Http.Resilience with a standard handler (retry, circuit breaker, timeout, rate limiting) without wiring Polly yourself:

builder.Services.AddHttpClient("inventory")
    .AddStandardResilienceHandler();

Interviews still expect Polly concepts; know both names if asked about “modern .NET.”

Design tips

  • Set HttpClient.Timeout and Polly timeout so they align; avoid unbounded retries.
  • Log each retry with correlation id; monitor circuit-open events.
  • Use IHttpClientFactory — do not create new HttpClient() per request.
  • Pair resilience with health checks and alerts on dependency failure rates.

Interview One-Liner

I attach Polly to IHttpClientFactory: retry with exponential backoff for 5xx/timeouts, circuit breaker to fail fast and prevent cascades, and timeouts so threads are not stuck. I only retry safe/idempotent calls and validate issuer-style config for production limits.

Interview Answer

1. Single Responsibility Principle (SRP)

A class should have only one responsibility or one reason to change.

    {

    }

    }
}

{
    public void Generate(Employee emp)
    {
        Console.WriteLine("Generating report");
    }

Benefit: easier maintenance, better testing, low coupling.

Software entities should be open for extension and closed for modification — add new functionality without changing existing code.

Bad example

public class PaymentProcessor
{

            Console.WriteLine("Processing credit card");

}

Problem: every new payment type requires modifying this class.

public interface IPayment

}

{
    public void Pay() => Console.WriteLine("Credit Card Payment");
}

{
    public void Pay() => Console.WriteLine("Paypal Payment");
}

public class PaymentProcessor
{
    public void ProcessPayment(IPayment payment) => payment.Pay();
}

Benefit: new payment types without changing old code.

3. Liskov Substitution Principle (LSP)

A derived class should be replaceable for its base class without affecting correctness.

Bad example (Bird)

public class Bird
{
    public virtual void Fly() => Console.WriteLine("Flying");
}

public class Ostrich : Bird
{
    public override void Fly()
    {
        throw new Exception("Ostrich can't fly");
    }
}

Problem: Ostrich breaks expected behavior of Bird.

Good example

public interface IBird { void Eat(); }
public interface IFlyingBird { void Fly(); }

public class Sparrow : IBird, IFlyingBird
{
    public void Eat() => Console.WriteLine("Eating");
    public void Fly() => Console.WriteLine("Flying");
}

public class Ostrich : IBird
{
    public void Eat() => Console.WriteLine("Eating");
}

Benefit: no unexpected behavior when substituting types.

4. Interface Segregation Principle (ISP)

Clients should not be forced to implement methods they do not use.

Bad example

public interface IWorker
{
    void Work();
    void Eat();
}

public class Robot : IWorker
{
    public void Work() => Console.WriteLine("Working");
    public void Eat() => throw new Exception("Robot doesn't eat");
}

Problem: Robot is forced to implement Eat().

Good example

public interface IWorkable { void Work(); }
public interface IEatable { void Eat(); }

public class Human : IWorkable, IEatable
{
    public void Work() => Console.WriteLine("Working");
    public void Eat() => Console.WriteLine("Eating");
}

public class Robot : IWorkable
{
    public void Work() => Console.WriteLine("Robot working");
}

Benefit: smaller, focused interfaces.

5. Dependency Inversion Principle (DIP)

High-level modules should not depend on low-level modules. Both should depend on abstractions.

Bad example

public class SqlServerDatabase
{
    public void Save() => Console.WriteLine("Saved to SQL Server");
}

public class EmployeeService
{
    private SqlServerDatabase db = new SqlServerDatabase();

}

Problem: EmployeeService is tightly coupled to SQL Server.

}

public class MongoDatabase : IDatabase
{
    public void Save() => Console.WriteLine("Saved to MongoDB");
}

public class EmployeeService
{
    private readonly IDatabase _database;

    public void SaveEmployee() => _database.Save();
}

1. Creational patterns

Singleton

Ensures only one instance of a class. Used for logger, configuration, cache, connection manager.

public sealed class Singleton
{
    private static Singleton _instance = null;
    private static readonly object _lock = new object();

    private Singleton() { }

    public static Singleton GetInstance()
    {
        lock (_lock)
        {
            if (_instance == null)
                _instance = new Singleton();
            return _instance;
        }
    }

    public void ShowMessage() => Console.WriteLine("Singleton instance");
}

// Usage
var obj1 = Singleton.GetInstance();
var obj2 = Singleton.GetInstance();
Console.WriteLine(obj1 == obj2); // True

Interview: private constructor, static instance, lock for thread safety. In ASP.NET prefer AddSingleton<T>().

Factory

Creates objects without exposing creation logic to the client.

// Without factory — tight coupling
// Car car = new Car();

public interface IVehicle { void Drive(); }

public class Car : IVehicle
{
    public void Drive() => Console.WriteLine("Driving Car");
}

public class Bike : IVehicle
{
    public void Drive() => Console.WriteLine("Driving Bike");
}

public class VehicleFactory
{
    public static IVehicle GetVehicle(string type) =>
        type == "Car" ? new Car() : new Bike();
}

IVehicle vehicle = VehicleFactory.GetVehicle("Car");
vehicle.Drive();

Creates families of related objects (e.g. WindowsButton + WindowsTextbox vs MacButton + MacTextbox).

Builder

Constructs complex objects step-by-step (e.g. Pizza with dough, cheese, topping) when constructors have too many parameters.

2. Structural patterns

Adapter

Makes incompatible interfaces work together (like a mobile charger adapter).

public interface ITarget { void Request(); }

public class Adaptee
{
    public void SpecificRequest() => Console.WriteLine("Specific Request");
}

public class Adapter : ITarget
{
    private readonly Adaptee _adaptee = new Adaptee();
    public void Request() => _adaptee.SpecificRequest();
}

Decorator

Adds behavior dynamically without modifying the class (Coffee + Milk + Sugar). Alternative to large inheritance hierarchies.

Facade

Simplified interface to a complex subsystem — e.g. StartComputer() internally starts CPU, memory, OS.

Proxy

Controlled access — authentication proxy, lazy loading, API gateway.

3. Behavioral patterns

Observer

One-to-many: when subject changes, dependents are notified (e.g. YouTube subscribers).

public interface IObserver { void Update(string message); }

public class Subscriber : IObserver
{
    public void Update(string message) => Console.WriteLine(message);
}

Strategy

Encapsulates interchangeable algorithms (Credit Card, UPI, PayPal).

public interface IPaymentStrategy { void Pay(); }

public class UPIPayment : IPaymentStrategy
{
    public void Pay() => Console.WriteLine("UPI Payment");
}

Command

Encapsulates a request as an object (remote control buttons).

State

Behavior changes with internal state (ATM: no card, card inserted, PIN entered).

Mediator

Centralizes communication (chat room server).

Repository pattern (.NET)

Separates data access from business logic.

public interface IEmployeeRepository
{
    Employee GetById(int id);
    void Save(Employee emp);
}

Benefits: cleaner architecture, easier unit testing, works with DI.

Unit of Work

Groups DB operations into a single transaction. EF Core DbContext is a common example.

MVC pattern

Separates Model, View, Controller — used in ASP.NET MVC / ASP.NET Core MVC.

Interview tip

If asked “Which design patterns have you used?” say: “I have mainly used Singleton, Repository, Factory, Dependency Injection, Strategy, and MVC in ASP.NET Core applications.”

Interview Answer

Design patterns are reusable templates in three groups: Creational (Singleton, Factory, Builder), Structural (Adapter, Decorator, Facade), Behavioral (Strategy, Observer, Command). In .NET I commonly use Repository, Factory, Strategy, Singleton via DI, and MVC — matching the problem, not forcing patterns everywhere.
